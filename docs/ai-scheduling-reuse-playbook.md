# AI 排课项目复用步骤清单

这份清单用于其他团队复用本项目方法。原则是：先确认口径，再标准化数据，再自动排课，最后用报告验收。

## 0. 准备共识

先确认一句话目标：

> 本轮要自动生成哪些班级、哪些日期范围、哪些课程类型的课表，并导入或发布到哪里？

必须先说清楚：

- 排课对象：班级、产品、考季、校区。
- 排除对象：计费类、服务类、已固定课表、无需自动排课的专项。
- 时间边界：排课开始、结束、节假日、特殊调休。
- 下游边界：是否导入 ERP，是否需要只读发布。

## 1. 数据准备

### 必备表

| 表 | 最小字段 |
| --- | --- |
| 班级表 | 班级 ID、班级名称、产品、考季、人数、开始/结束日期、套班编码 |
| 产品表 | 产品 ID、产品名称、产品线、子产品、产品体系 |
| 产品课程表 | 产品 ID、科目、阶段、模块、课程分组、总课时、阶段顺序 |
| 产品窗口规则表 | 产品 ID、季节窗口、授课形式、可排星期、可排时段、单次连续课时 |
| 老师安排表 | 班级 ID、科目、阶段、课程分组、老师 ID、老师姓名 |
| 教师表 | 老师 ID、姓名、主教学科目、状态 |
| 教室表 | 教室 ID、教室名、教学区、容量 |
| 冲突组表 | 冲突组 ID、班级 ID |

### 强烈建议补充

- 历史已排课明细：用于抵扣课时和学习老师安排。
- 锁定课表：专业课、人工已定课、不可移动课。
- 班级老师安排中的共享课表关系：维护实际排课班级和共享班级。
- 全局不可排日期：节假日、调休、业务停课。
- 教学区关联：用于老师同日跨区通勤判断。

## 2. 字段治理

关键字段不要混用：

- `start_date/end_date`：排课窗口。
- `first_lesson_date/first_lesson_period`：真正固定首课。
- `locked_scheduled_lessons`：不可移动的已定课。
- `class_conflict_groups`：同一学生群体或套班互斥。
- `product_schedule_rules`：产品或课程粒度的日期、星期、时段规则。

检查原则：

- 所有 ID 尽量稳定，不用显示名称做唯一键。
- 老师、教室、班级 ID 要能回写到下游系统。
- 产品课程模板要能解释每个班为什么需要这些课时。

## 3. 规则拆分

### 硬约束

硬约束不通过就不能发布：

- 同一老师同一课节不能上两个班。
- 同一班级同一课节不能有两节课。
- 同一教室同一课节不能重复占用。
- 冲突组内班级不能同时上课。
- 课程必须在产品/班级/老师允许的时间内。
- 锁定课表不能被移动。
- 全局不可排日期不能排课。

### 软约束

软约束用于评分、排序、优化：

- 周课量均衡。
- 分科目周分布均衡。
- 同科目不要连续过多天。
- 英语和政治尽量交替。
- 避免同班同科同日 8 小时。
- 老师同日尽量同校区或同区域。
- 教室容量不足作为提醒或门禁，取决于业务口径。

## 4. 首轮运行

推荐流程：

1. 上传源数据。
2. 生成预填模板。
3. 下载缺口清单。
4. 人工补齐产品映射、合班、老师安排。
5. 跑上传前校验。
6. 校验通过后正式运行。

不要跳过上传前校验。校验阶段应该只读，不改正式数据。

## 5. 审计与修复

### 覆盖审计

目标：

- 自动排课需求缺口班级数为 0。
- 自动排课需求缺口课时为 0。
- 额外展示课表有来源解释。

### 质量审计

先看高优先级：

1. 老师同日远距离跨区。
2. 同班同科同日课量过重。
3. 活跃周空白。
4. 分科目周分布极端不均。
5. 老师连续上课天数偏多。

修复顺序：

1. 不破坏硬约束。
2. 不造成需求覆盖缺口。
3. 优先处理影响最大的一批班级或老师。
4. 每轮修复后重新跑覆盖审计和质量审计。

## 6. ERP 或下游系统对齐

需要单独建立下游检查：

- 字段是否齐全。
- ID 是否能匹配。
- 主班/从班是否符合导入口径。
- 已存在课次如何更新或跳过。
- 导入失败是否有分类统计。
- 是否需要差异导入表，而不是全量覆盖。

发布前要求：

- 映射/数量异常为 0。
- 导入字段缺口为 0。
- 调整课次和变化字段可解释。

## 7. 分享与发布

对外分享建议只读化：

- 管理后台只在本机或内网使用。
- 跨部门查看使用只读站。
- 只读站只提供课表查看、CSV 下载、报告下载和健康检查。
- 不开放保存、导入、重排、数据目录访问。

## 8. 每轮迭代记录

建议每轮都记录：

```text
目标：
输入数据版本：
调整脚本和参数：
调整前备份：
输出课表：
覆盖审计结果：
质量审计结果：
ERP 检查结果：
仍需人工确认：
下一步：
```

## 9. 复用时最容易踩坑的点

- 把业务系统产品名当成稳定产品 ID。
- 用班级名称猜合班关系。
- 没上传历史课表就直接排剩余课时。
- 把专业课固定课表和公共课自动排课混在一起移动。
- 把首课日期误当成窗口开始日期。
- 只检查是否排满，不检查老师、教室、互斥和 ERP。
- 只修眼前一张课表，不保留脚本、报告和备份。

## 10. 最小可复用闭环

如果其他团队只想先做一个小范围试点，最小闭环是：

1. 选 5-10 个班级。
2. 准备班级、产品课程、老师、教室、时间槽、冲突组。
3. 跑一次预检，补齐缺失字段。
4. 排出 CSV/HTML。
5. 跑覆盖审计。
6. 跑质量审计。
7. 人工确认 3-5 个典型问题。
8. 再扩大范围。
