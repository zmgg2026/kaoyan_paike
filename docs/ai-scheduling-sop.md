# AI 排课项目 SOP

> 适用范围：需要把业务班级、产品课程、教师、教室、历史课表和排课规则整理成可校验、可回退、可发布课表的排课项目。
> 核心原则：不要把 AI 排课理解成“一键生成课表”。稳定交付依赖三件事：口径清楚、数据可跑、结果有证据。

## 1. 交付目标

一次完整排课交付必须产出：

- 标准化输入数据：年度排课窗口、课节、教学区、教室、教师、产品、课程课时、产品窗口排课规则、班级、班级排课窗口、老师安排、冲突组、锁定课表。
- 排课结果：课表明细 CSV 和可视化 HTML。
- 验收证据：导入报告、覆盖审计、质量审计、ERP 对齐结果。
- 回退证据：运行前备份、失败目录、调整记录和可复跑脚本。
- 分享版本：只读查看页或自包含 HTML，不开放保存、导入、重排入口。

## 2. 分工

| 角色 | 负责内容 | 关键确认 |
| --- | --- | --- |
| 业务负责人 | 排课范围、排除项、例外口径、最终发布确认 | 哪些班级进入本轮，哪些不排，哪些课表不能动 |
| 教学/教务 | 老师安排、合班、固定课表、特殊日期 | 老师是否唯一，合班是否完整，固定课表是否锁定 |
| 数据维护 | 基础资源、产品课程、产品规则、班级窗口、互斥组 | ID 是否稳定，字段是否能回写 ERP |
| AI/程序执行 | 数据转换、预检、排课、审计、报告、只读发布 | 每轮是否可复跑、可回退、可解释 |

## 3. 标准流程

### Step 0：冻结本轮范围

先确认一句话目标：

> 本轮要为哪些班级、哪些日期范围、哪些课程类型生成课表，并导入或发布到哪里？

必须明确：

- 排课对象：项目、产品、考季、校区、班级。
- 排除对象：计费类、服务类、无需自动排课专项、已固定课表。
- 时间边界：最早可排日期、最晚结课日期、节假日、调休和停课。
- 下游边界：是否导入 ERP，是否需要只读发布。

### Step 1：准备基础数据

先收齐 16 张基础表：

| 表 | 作用 | 必填程度 |
| --- | --- | --- |
| 课节表 | 定义可排日期、星期、上午/下午/晚上、课节时长和是否可用 | 必填 |
| 教学区表 | 定义教学区、容量、区域标签、地址 | 必填 |
| 教室表 | 定义具体教室、所属教学区、容量和可用状态 | 必填 |
| 教师基础信息表 | 定义教师这一类基础资源的唯一 ID、教学归属和用工状态 | 必填 |
| 教师不可排日期时段表 | 只记录老师不可排课的日期/时段例外；兼职长期限制、全职临时请假、培训、会议都填这里 | 按需必填 |
| 产品管理表 | 定义产品 ID、产品线、子产品、体系、科目和产品可选阶段 | 必填 |
| 年度排课窗口表 | 定义可直接引用的年度窗口，例如 2026暑假、2026秋季、2027寒假 | 必填 |
| 产品课程课时表 | 定义每个产品要上什么课、多少小时和阶段模块顺序 | 必填 |
| 产品窗口排课规则表 | 定义产品在寒假/春季/暑假/秋季这类季节窗口内，星期、时段和每周课时怎么排 | 强烈建议 |
| 班级基础信息表 | 定义班级自身信息、产品 ID、考季、人数和排课范围 | 必填 |
| 班级排课窗口表 | 定义班级在每个实际排课窗口内的边界和窗口级资源要求 | 必填 |
| 班级老师安排表 | 定义班级在科目、阶段和课程组上的老师，以及合班时共享哪个实际排课班级的课表 | 必填 |
| 班级排课互斥关系表 | 定义同一学生群体或报名组合不能占用同一课节 | 强烈建议 |
| 锁定课表 | 定义已人工排定且不能移动的课 | 按需必填 |
| 教学区通勤关系表 | 定义跨区距离和通勤关系 | 质量优化 |
| 全局停课日期表 | 定义所有产品都不能排课的日期 | 按需必填 |

数据整理口径：

- ID 用稳定编码，不用显示名称做唯一键。
- 课节表默认生成 2026-07-01 到 2027-12-21 的明细；每天默认上午 2 节、下午 2 节、晚上 1 节。
- 课节表必须维护 `weekday`、`is_usable`、`availability_source`、`unavailable_reason`：如果某一天所有产品都不能排课，当天所有课节都标记为不可用。
- 默认整天不可用规则：春季窗口每周一不可用，暑假窗口每周日不可用，秋季窗口每周一不可用，寒假窗口每周日不可用。
- 法定节假日当天默认不可用；2027 年调休安排未发布前，只按法定节日当天预填，不把调休连休整段直接封停。
- 后续人工指定某天不能排课时，把当天相关课节 `is_usable` 改为 `否`，`availability_source` 改为 `人工指定不可排`，并在 `unavailable_reason` 写清原因。
- 年度排课窗口表直接维护本轮可引用的实际排课窗口：例如 `2026暑假`、`2026秋季`、`2027寒假`；寒假为 1-2 月，春季为 3-6 月，暑假为 7-8 月，秋季为 9-12 月。
- 教学区表不维护楼栋、楼层和默认可用课节；资源可用性由课节表、产品窗口排课规则和班级排课窗口共同决定。
- 教室表不维护楼栋、楼层和设备字段；排课只使用教室 ID、所属教学区、容量、教室类型和启用状态。
- 教师基础信息表和教室表一样属于基础资源；老师是否能排课、是否冲突，后续都通过教师 ID 关联判断。
- 教师不可排日期时段表不是教师可用时间表，只记录不可排例外；大部分全职老师不需要逐条维护可用时间，但全职老师请假、培训、会议等临时不可排也填这里。兼职长期限制、阶段性不可排和临时不可排都按“一行一条不可排规则”填写，同一个老师可以有多条记录；项目和用工类型从教师基础信息表按教师 ID 关联，不在本表重复维护；不可排规则 ID 可按规则自动生成。
- 产品管理表维护产品可选阶段；班级基础信息表的班级适用阶段从产品可选阶段中勾选。
- 产品课程表回答“要上什么、多少小时、阶段模块顺序”。
- 产品窗口排课规则表回答“在寒假/春季/暑假/秋季这类季节窗口里，周几能排、上午/下午/晚上能不能排、单次连续几小时、每周上下限”；排课时用班级实际窗口里的 `season_window_id` 匹配产品规则。
- 同一半天白天连续排 4 小时、拆成 2 节课时，必须安排同一个老师。
- 班级基础信息表的“班级实际排课窗口ID列表”优先按首课日期、缺失时按开始日期，与实际结课日期和产品允许季节窗口求交集生成，值直接来自年度排课窗口表，例如 `2026暑假|2026秋季`。
- 排课窗口 ID 使用“年度 + 窗口名”，例如 `2026秋季`；另用 `window_order` 按年份和窗口起始月份排序，例如 `202609`，避免和财季口径混淆。
- 班级排课窗口表按“班级 * 实际排课窗口”生成对应数量的数据行，回答“这个班级在这个年度窗口内，最早/最晚从哪天哪个时段排；如果是面授窗口，还可以单独指定教学区和教室”；`该窗口是否纳入排课=否` 时，保留记录但本轮不自动生成需求。
- 班级自己的排课窗口可以比年度排课窗口更窄；寒暑营、暑假营等面授窗口的日期、时段、教学区和教室统一落在班级排课窗口中，不再另设同类控制表。如需跨窗口排课，必须在产品规则或班级排课窗口备注里说明原因。
- 老师安排维护到 `班级 + 科目 + 阶段 + 课程组`，不依赖业务导出里的文本老师名；`班级课表处理方式=本班实际排课` 时本班承载课表并填写老师 ID，`班级课表处理方式=共享实际排课班级` 时填写实际排课班级 ID，共享该班对应课次的时间、老师和教室信息；本安排有特殊时间要求时，填 `本安排额外时间要求`，老师本人的全局不可排时间统一填教师不可排日期时段表。
- 班级排课互斥关系表一行表示一组不能同课节上课的班级，`班级ID列表` 用 `|` 分隔；互斥组可来自报名套班、人工补充或历史课表识别，停用时保留记录但不参与本轮冲突检查。
- 锁定课表只记录已人工确定不能移动的课；历史课表用于抵扣已排课时、学习老师安排、识别合班候选，不能用锁定课表或最终维护页课表替代。

产品排课规则按两层整理：

| 层级 | 记录粒度 | 说明 |
| --- | --- | --- |
| 产品窗口规则 | 产品 + 季节窗口 | 定义开课后在寒假、春季、暑假、秋季分别可排的星期、时段、单次课时、每日上限和每周上下限 |
| 班级排课窗口 | 班级 + 实际排课窗口 | 根据班级实际开结课日期生成 `2026暑假`、`2026秋季` 这类含年份窗口，并定义最早/最晚排课日期时段 |

示例口径：

- 无忧秋：开课后，秋季/寒假/春季窗口每周周六、周日白天可排，每个班级每天不超过 4 小时；暑假窗口每周周一到周六白天可排，每天不超过 4 小时。
- 无忧寒：开课后，寒假窗口每周周一到周六白天可排，每天不超过 4 小时；春季窗口每周周二到周五晚上可排，每次 2 小时；暑假窗口每周周一到周六白天可排；秋季窗口每周周二到周六晚上可排，每次 2 小时。

若从业务系统导出开始正式试运行，另备控制、历史和映射表：

| 表 | 作用 |
| --- | --- |
| 历史已排课明细表 | 抵扣已排课时、学习老师安排、生成合班候选；没有独立历史源时保持为空 |
| ERP产品对应表 | 将本地产品与 ERP 标准课程产品关联，保证后续能回写和核对 |

### Step 2：生成预填模板

使用排课运行维护页或 pipeline 生成模板和缺口清单。

本地后台：

```bash
./scripts/start_admin.sh
```

打开 `http://127.0.0.1:8765`，进入“排课运行维护”：

1. 上传业务班级导出、历史已排课明细和可选标准数据。
2. 下载 `formal_launch_template_<timestamp>.xlsx` 或 CSV 模板包。
3. 查看 `template_report_<timestamp>.md` 和缺口清单。

预填模板常见输出：

- 产品目录参考。
- 业务班级筛选结果。
- 历史学习到的老师安排。
- 已排课时抵扣。
- 疑似合班候选。
- 产品映射、老师安排、合班共享课表关系等缺口。

### Step 3：补齐人工确认项

优先补齐会阻断运行的表：

1. ERP产品对应：本地产品映射到 ERP 标准课程产品。
2. 产品排课规则：产品在各窗口期的可排星期、时段、单日上限和单次课时。
3. 班级排课窗口：班级在各窗口期的最早/最晚可排日期、时段和窗口级教学区/教室。
4. 班级老师安排：科目、阶段和课程组粒度的老师，以及合班共享课表关系。
5. 班级排课互斥关系：同一报名组合或学生群体不能同时上的班级组。
6. 锁定课表：已经人工确认不能移动的课。

确认边界：

- `产品体系=常规体系` 默认排课。
- `专项体系` 需要白名单纳入。
- `计费体系` 不进入自动排课。
- 有合班关系但没有在班级老师安排表中明确实际排课班级时，必须先补齐。

### Step 4：上传前校验

上传前校验只做识别、转换和缺口检查：

- 不备份。
- 不写入正式 `data/`。
- 不排课。

校验不通过时，先看报告，不要直接进入正式运行。

重点处理：

- 产品映射缺失。
- 老师安排缺失。
- 合班共享课表关系缺失。
- 班级缺少开课/结课日期。
- 班级缺少窗口期最早/最晚排课边界。
- 产品缺少窗口期排课规则。
- 科目缺少窗口期周课时上下限。
- 课程总课时不能被产品窗口规则里的单次连续课时整除。
- 老师、教室、班级 ID 不能回写。
- 锁定课表字段缺失。

### Step 5：正式运行

校验通过后再正式运行。正式运行前系统必须备份当前 `data/`。

运行方式：

```bash
./scripts/run_pipeline.sh incoming
```

或在后台“排课运行维护”上传同一批补齐文件运行。

正式运行应产出：

- `data/scheduler_input_draft.json`
- `outputs/schedule_<timestamp>.csv`
- `outputs/schedule_<timestamp>.html`
- `outputs/import_report_<timestamp>.md`
- `outputs/backups/<timestamp>/`
- 参考文件和缺口文件。

### Step 6：覆盖审计

覆盖审计回答：本轮要排的课是否排满。

发布前要求：

- 自动排课需求班级数清楚。
- 需求缺口班级数为 0。
- 需求缺口课时为 0。
- 额外展示课表能解释来源。

### Step 7：质量审计

质量审计回答：课表是否明显不合理。

先看高优先级问题：

- 老师同日远距离跨区。
- 同班同科同日课量过重。
- 活跃周空白。
- 分科目周分布极端不均。
- 老师连续上课天数偏多。

处理顺序：

1. 不破坏硬约束。
2. 不造成覆盖缺口。
3. 优先修影响最大的一批班级或老师。
4. 每轮修复后重新跑覆盖审计和质量审计。

### Step 8：ERP 对齐

ERP 对齐是一条单独验收链，不是最后简单导出。

需要检查：

- 班级、老师、教室、课程字段是否齐全。
- ID 是否能匹配。
- 主班/从班是否符合导入口径。
- 已存在课次如何更新、跳过或反向调整。
- 导入失败是否有分类统计。
- 差异导入表能否解释每条变化。

发布前要求：

- 映射/数量异常为 0。
- 导入字段缺口为 0。
- 调整课次和变化字段可解释。

### Step 9：只读发布

只读发布必须和管理后台隔离。

本地预览：

```bash
python3 schedule_publish_server.py --hash-password
export SCHEDULE_VIEWER_USERNAME="schedule-viewer"
export SCHEDULE_VIEWER_PASSWORD_HASH="上一步生成的哈希"
export SCHEDULE_VIEWER_SECRET_KEY="$(python3 -c 'import secrets; print(secrets.token_urlsafe(48))')"
PORT=8780 ./scripts/start_schedule_publish.sh
```

只读站只开放：

- `/schedule`
- `/download/schedule.csv`
- `/download/report.md`
- `/healthz`

不开放：

- 保存。
- 导入。
- 重排。
- 后台管理接口。

## 4. 发布门禁

发布前只看 5 个信号：

| 信号 | 绿灯标准 |
| --- | --- |
| 不撞车 | 老师、班级、教室、互斥组、锁定课表硬冲突为 0 |
| 不漏课 | 缺口班级数为 0，缺口课时为 0 |
| 有结论 | 高优先级质量问题已处理或说明原因 |
| 能导入 | ERP 字段缺口为 0，映射和数量异常为 0 |
| 只读分享 | 分享页无保存、导入、重排等写入入口 |

## 6. 常见失败与处理

| 现象 | 常见原因 | 处理 |
| --- | --- | --- |
| 产品映射缺失 | 业务系统产品编码没有映射到本地排课产品 | 补 `business_product_mappings`，必要时按班级关键词拆分 |
| 老师安排缺失 | 业务导出老师名不能替代正式安排 | 补 `class_teacher_assignments`，用唯一老师 ID |
| 合班阻塞 | 业务表有合班关系，但没有实际排课班级 | 补 `12_班级老师安排表` 的 `class_schedule_mode` 和 `actual_scheduled_class_id` |
| 找不到可行课表 | 可用课节不足、老师/教室/互斥过紧 | 扩大时间窗口、核对教师不可排日期时段、调整教室或冲突组 |
| 质量问题多 | 硬约束已满足，但体验规则未优化 | 先修高优先级，修后重新审计 |
| ERP 导入失败 | 字段缺口、ID 不匹配、已有课次冲突 | 分类失败原因，生成重导或差异调整表 |
